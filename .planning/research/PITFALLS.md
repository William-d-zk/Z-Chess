# Pitfalls Research: 边缘算力调度与IM平台

## 常见错误与预防

### 1. 任务调度相关

#### 任务碎片化导致调度开销过大

**问题：** 大量小任务（<1ms执行时间）导致调度开销超过计算本身。

**预警信号：**
- 调度延迟 > 任务执行时间
- 边缘节点频繁上报心跳但少有任务完成
- 调度器CPU成为瓶颈

**预防策略：**
- 任务最小粒度限制（如 >10ms）
- 任务批量打包分发
- 边缘端本地任务队列

#### 结果汇总死锁

**问题：** 协调者等待所有结果，但某些节点永久失败。

**预警信号：**
- 任务长时间处于 PARTIAL_COMPLETE
- 监控发现某节点心跳中断

**预防策略：**
- 超时机制（任务最大执行时间）
- 冗余算力自动接管
- 优雅降级（部分结果可接受时提前汇总）

#### 调度策略与网络拓扑不匹配

**问题：** 分发型任务发送到高延迟或不稳定链路。

**预警信号：**
- 任务分发成功但执行超时率高
- 某些区域节点任务失败率明显偏高

**预防策略：**
- 网络拓扑感知调度
- 就近分发原则
- 分组隔离（高延迟链路不参与实时任务）

### 2. 边缘计算相关

#### 边缘节点算力上报不准确

**问题：** 节点注册时上报算力与实际不符，导致分配不合理。

**预警信号：**
- 同等任务在不同节点执行时间差异大
- 节点实际CPU/内存与注册信息不符

**预防策略：**
- 调度前探针任务（5%样本）
- 动态算力评估
- 差异化SLA（不保证精确分配）

#### 边缘存储与计算争抢资源

**问题：** 存储密集型任务与计算争抢IO/内存。

**预警信号：**
- 任务执行时节点CPU高但实际计算效率低
- 本地存储IO成为瓶颈

**预防策略：**
- 资源隔离（Docker cgroups）
- 任务类型标记，异构调度
- 存储计算分离部署

### 3. IM服务相关

#### 消息顺序与因果一致性

**问题：** 群组消息乱序，用户体验差。

**预警信号：**
- 用户反馈消息顺序混乱
- 消息ID递增但时间戳跳跃

**预防策略：**
- 序列号机制
- 客户端本地重排序
- 因果一致性（同一会话消息有序）

#### 消息洪泛导致服务过载

**问题：** 设备状态变更触发大量IM推送，消息风暴。

**预警信号：**
- WebSocket连接数正常但消息延迟飙升
- 消息队列积压

**预防策略：**
- 消息合并（状态变更批量推送）
- 客户端订阅过滤
- 消息限速（单用户/群组）

#### 消息与设备操作时序问题

**问题：** 用户在IM看到设备状态更新，但实际设备动作还未执行。

**预警信号：**
- 用户反馈操作无效（操作时设备实际离线）
- 设备状态与IM显示不一致

**预防策略：**
- 操作命令独立通道（优先级高于消息）
- 乐观更新 + 最终一致性确认
- 操作结果单独通知

### 4. 系统级问题

#### 网络分区导致脑裂

**问题：** RAFT集群网络分区，每侧都认为自己是Leader。

**预警信号：**
- 调度任务被重复执行
- 结果被多次上报
- 集群出现多个Leader日志

**预防策略：**
- RAFT的majority机制保活性
- 分区检测与自动恢复
- 操作幂等性设计

#### 百万终端接入的连接管理

**问题：** MQTT Broker连接数成为瓶颈。

**预警信号：**
- Z-Pawn Broker连接数接近上限
- 消息投递延迟随连接数增加而上升

**预防策略：**
- 连接复用（HTTP/2/WebSocket）
- 分层Broker架构
- 边缘中转节点

### 5. 延迟保障相关

#### 40ms延迟无法保证

**问题：** 控制指令经调度链路延迟超过40ms。

**预警信号：**
- 端到端延迟监控 > 40ms
- 用户感知控制指令"卡顿"

**预防策略：**
- 控制指令专用快速通道（绕过调度器）
- 边缘预置指令缓存
- 就近执行（边缘节点直接控制本地设备）

#### 不同优先级任务互相干扰

**问题：** 低优先级批处理任务占用资源，影响高优先级实时任务。

**预警信号：**
- AI推理任务延迟在批处理期间明显增加
- 设备控制指令延迟不稳定

**预防策略：**
- 资源预留（保证高优先级任务资源）
- 优先级抢占调度
- 资源隔离（不同类型任务在不同容器）
